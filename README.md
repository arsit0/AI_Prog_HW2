##1.任务拆解与AI协作策略
这次作业要求我开发一个“学生信息与考场管理系统”，而且不只是把功能写出来就行，还要求用面向对象、异常处理、静态方法、版本控制这些内容，最后还要写README记录开发过程。刚开始我如果直接让AI“一把梭”生成完整代码，其实很容易翻车，因为生成出来的东西不一定完全符合题目要求，看起来也很像复制粘贴的成品。所以我没有直接一步到位，而是把整个任务拆成了几个小步骤，一步一步做。

我一开始先把题目要求捋清楚，先确认这个系统最核心的4个功能：按学号查找学生信息、随机点名、生成考场安排表、生成准考证文件。然后我先搭项目结构，把 `main.py`、`student.py`、`exam_system.py`、`README.md` 和学生名单文件先准备好。接着我先单独完成 `Student` 类，因为这个类比较基础，主要负责保存单个学生的信息，同时实现 `__init__` 和 `__str__`，这样后面打印学生对象时会方便很多。

在这之后，我再去完成 `ExamSystem` 类。这个类算是整个系统的“大脑”，负责读取学生名单、按学号查找、随机点名、生成考场安排表、生成准考证这些功能。我没有一次把所有功能全塞进去，而是按要求一个一个加。先做读取文件，再做查找，再做随机点名，再做考场安排，最后做准考证。这样做的好处很明显：每一步都比较清楚，哪里出错了也好找，不会写着写着把自己绕晕。

整个过程中，我把AI更多当成一个“会给建议的辅助工具”，不是当成替我交作业的“代写机器”。具体来说，AI主要帮我做了三件事：第一，帮我把题目要求拆开，让我知道先做什么后做什么；第二，在我写代码或者改代码时，帮我检查逻辑有没有漏洞；第三，在我报错的时候，帮我根据报错信息定位问题，比如缩进错误、路径错误、输入异常这些。真正决定怎么改、怎么测、怎么提交，还是我自己一步一步完成的。

另外，这次我还专门按“小步提交”的原则做版本控制。也就是说，我不是等所有代码写完了再一次性提交，而是每完成一个阶段就 commit 一次。这样做虽然麻烦一点，但看起来更像真正的开发过程，也更符合老师要求。整体做下来，我最大的感受是：把大任务拆小之后，事情一下就没那么吓人了，每一步都去矫正ai给的答案，会让结果完全准确

##3.Debug与异常处理记录
###报错类型/漏洞现象
这次开发过程中，我印象最深的一个报错是 `IndentationError`，把错误信息喂给了ai确认了烦人的缩进错误在哪里。它出现的时候，程序根本跑不起来，终端会直接提示某个函数定义后面没有正确缩进，或者某一行出现了意外缩进。
大概是因为粘贴的时候出了问题，不是自己写的代码，让里面加东西的时候会很崩溃
###原因分析

这个问题主要出现在我给 `exam_system.py` 增加 `random_call` 方法的时候。因为我一边参考功能要求一边手动调整代码，有一段函数体的缩进没有和前面的类方法保持一致。Python不像有些语言靠大括号区分代码块，它完全靠缩进判断层级，所以只要某一行多缩进了几个空格，或者少缩进了几个空格，解释器就会当场翻脸。
###解决过程

我先根据终端给出的报错信息定位到具体行数，然后回到对应文件去检查那一段代码。后来发现问题不是逻辑写错了，而是函数体里面的文档字符串、`if` 判断、`raise` 语句和 `return` 语句的缩进层级不统一。于是我把这一整个函数重新整理了一遍，保证方法定义下面的内容统一缩进4个空格，方法内部的判断再往里缩进4个空格。改完之后重新运行，程序就恢复正常了。
除了这个缩进问题，我还遇到过路径相关的小问题。比如一开始在终端里运行 `python main.py` 的时候，当前目录不对，Python找不到文件，就会提示无法打开 `main.py`。后来我学会了先切到项目根目录再运行，这个问题也就解决了。虽然这些问题看起来不算大，但它们很真实，也让我更明白“程序能不能跑起来”不仅取决于逻辑，还取决于很多很细的细节。
###异常处理设计

在异常处理方面，我主要做了两类设计。第一类是读取文件时的异常处理。在加载学生名单时，我使用了 `try-except` 捕获 `FileNotFoundError`，这样即使文件缺失，程序也不会直接崩掉，而是给出明确提示。第二类是用户输入时的异常处理，尤其是随机点名功能。因为用户输入的人数需要先用 `int()` 转成整数，所以如果输入了 `abc` 这种内容，就会触发 `ValueError`。我在主程序里用 `try-except ValueError` 捕获了这个问题，同时在业务方法里也主动判断人数是否合法，比如人数小于等于0，或者超过学生总人数时，都会抛出异常并提示用户重新输入。这样一来，程序不会因为一个乱输入就直接死机，整体交互体验也会好很多。

##4.人工代码审查（Code Review）
下面这段是我对“生成考场安排表”这个核心方法做的人工注释。这个部分我没有只看结果，而是把每一行在干什么都重新梳理了一遍。写完之后我自己回头再看，基本已经能讲清楚这段代码的工作流程了。
```python
def generate_exam_arrangement(self):
    """生成考场安排表文件，并返回输出文件路径"""
    #如果当前没有学生数据，就没法安排考场，所以直接抛出异常
    if not self.students:
        raise ValueError("当前没有学生数据，无法生成考场安排表。")
    #先复制一份学生列表，避免直接把原来的 students 顺序打乱
    shuffled_students = self.students[:]
    #对复制出来的列表进行随机打乱
    random.shuffle(shuffled_students)
    #每次重新生成安排表之前，先把旧结果清空
    self.exam_arrangement = []
    #enumerate(..., start=1) 的作用是从1开始编号，正好可以作为座位号
    for seat_number, student in enumerate(shuffled_students, start=1):
        #把“座位号+学生对象”保存成一个元组，放进考场安排列表中
        self.exam_arrangement.append((seat_number, student))
    #获取当前脚本所在目录，这样可以保证文件生成在项目根目录里
    base_dir = os.path.dirname(os.path.abspath(__file__))
    #拼接出“考场安排表.txt”的完整路径
    output_path = os.path.join(base_dir, "考场安排表.txt")
    #获取当前时间，并按题目要求格式化
    now_str = datetime.now().strftime("%Y-%m-%d %H:%M:%S")
    #以写入模式打开文件，没有文件就自动创建
    with open(output_path, "w", encoding="utf-8") as file:
        #第一行写生成时间
        file.write(f"生成时间：{now_str}\n")
        #第二行写表头，方便查看
        file.write("座位号\t姓名\t学号\n")
        #把每个学生的座位号、姓名、学号逐行写进去
        for seat_number, student in self.exam_arrangement:
            file.write(f"{seat_number}\t{student.name}\t{student.student_id}\n")
    #返回输出文件路径，主程序里可以直接提示用户
    return output_path
